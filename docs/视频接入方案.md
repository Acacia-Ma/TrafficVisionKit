# STM32 UDP 视频接入方案

## 1. 总体架构

```
STM32 (ATK-MC5640 + MO395Q)
  │  UDP JPEG 分片包（~10-15 fps，每包 ≤1016 字节）
  ▼
UDPReceiver  ─── asyncio DatagramProtocol ───  backend/udp/receiver.py
  │
  │  源 IP → 数据库查询 device_id
  │  首包触发 PipelineManager.on_device_connected()
  ▼
DeviceAssembler  ──────────────────────────────  backend/udp/assembler.py
  │
  │ ┌──────────────────────────────────────────┐
  │ │  Level 1：分片装配窗口（500 ms 超时）     │
  │ │  收齐所有 chunk → 验证 FFD8...FFD9        │
  │ │  未收齐 → 超时后静默丢弃                  │
  │ └──────────────────────────────────────────┘
  │
  │ ┌──────────────────────────────────────────┐
  │ │  Level 2：抖动缓冲（depth=3 帧）          │
  │ │  drain task 以 12 fps 匀速送入 raw_queue  │
  │ │  缓冲空 → 跳过本周期（不重播旧帧）        │
  │ │  缓冲满 → 丢最旧帧，保留最新内容          │
  │ └──────────────────────────────────────────┘
  │
  ▼
ctx.raw_queue  (maxsize=2，已有的 pipeline 入口，与 TCP 路径完全相同)
  │
  ▼
inference_loop → dispatch_loop → ws_push_loop → WebSocket → 前端
```

---

## 2. 为什么选 UDP？

| 指标 | TCP | UDP |
|------|-----|-----|
| 连接建立 | 三次握手，~ms 延迟 | 无连接，即发即到 |
| 丢包处理 | 自动重传（增加延迟） | 直接丢帧（实时优先） |
| 串口阻塞 | 需要等 ACK | 不需要 |
| 嵌入式资源占用 | 高（TCP 状态机） | 低（MO395Q 直接支持） |
| 视频流适合度 | 一般 | 高（与 RTSP/WebRTC 一致） |

## 3. 平滑无模糊方案详解

### 3.1 模糊 / 花屏（根因：JPEG 不完整）

**问题**：UDP 丢包导致某帧的 chunk 缺失，拼出来的 JPEG 截断，解码出花屏。

**解决**：`DeviceAssembler._is_valid_jpeg()` 强制校验：
- 头 2 字节必须是 `FF D8`（SOI）
- 尾 2 字节必须是 `FF D9`（EOI）

任何一条不满足 → 整帧丢弃，永远不进入 raw_queue。

### 3.2 重影（根因：同一帧被重复显示）

**问题**：若某帧完整，下一帧缺失，系统可能把上一帧反复推送给前端。

**解决**：抖动缓冲的 drain task 采用 **跳过策略**：缓冲空时什么都不做，不重播旧帧。`_last_emitted_id` 记录已输出的最新 frame_id，保证每个 ID 最多出现一次。

### 3.3 卡顿 / 不均匀（根因：UDP 到达具有突发性）

**问题**：STM32 处理完一帧后立刻发送 15-20 个 UDP 包，到达时间集中；下一帧间隔可能较长，导致前端出现"一瞬间多帧 + 空窗期"的卡顿感。

**解决**：Level 2 抖动缓冲 + 匀速 drain：
```
完整帧到达 → 放入 jitter_buf（depth=3）
drain task 每 1/12 s 取出一帧 → raw_queue → YOLO → WS
```
效果：输出端看到的是 12 fps 的匀速流，即使 STM32 侧是 8 fps 或 15 fps 的不规律流，前端也始终平滑。

### 3.4 超时清理（根因：设备重启导致旧装配状态残留）

- 每帧的 chunk 装配有 **500 ms 截止时间**。
- 超过截止时间仍未收齐 → 丢弃，不等待。
- 设备 **30 s** 无数据 → watchdog 触发 `on_device_disconnected()`，状态完全清零。

## 4. 关键参数说明（可在 assembler.py 顶部调整）

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `CHUNK_DEADLINE_S` | 0.5 s | 等待分片超时，超时丢帧。局域网建议 0.3-0.5 s |
| `JITTER_DEPTH` | 3 帧 | 抖动缓冲深度。越大越平滑，延迟越高 |
| `TARGET_FPS` | 12 fps | drain 输出速率。建议与 STM32 实际帧率接近 |
| `MAX_PENDING` | 8 帧 | 同时追踪的最大未完成帧数 |
| `UDP_IDLE_TIMEOUT_S` | 30 s | 设备离线判定超时 |

## 5. 网络配置要求

| 项目 | 要求 |
|------|------|
| STM32 IP | 与数据库 `devices.ip_address` 一致 |
| PC/服务器 IP | `192.168.100.50`（可在 STM32 demo.c 中修改） |
| UDP 端口 | `8080`（.env 中 `UDP_PORT=8080`） |
| 网络 | 同一局域网直连或同交换机，避免经 NAT |

---

## 6. 数据库配置

无需修改表结构。只需在 `devices` 表中为 STM32 设备配置正确的 `ip_address`：

```sql
-- 示例：将 STM32 的 IP 192.168.100.100 注册为设备 1
UPDATE devices SET ip_address = '192.168.100.100' WHERE id = 1;
```

或通过管理接口：`PUT /api/devices/{id}` 更新 `ip_address` 字段。

## 7. 启动验证

后端启动后应看到：

```
[UDP] listening on 0.0.0.0:8080
```

STM32 开机并连接网线后，应依次看到：

```
[UDP] ✓ device_id=1 IP=192.168.100.100 connected
[UDP] device 1 drain task started (12.0 fps)
```

## 8. 调试技巧

在 `assembler.py` 中将 logger 级别临时改为 DEBUG 可看到每帧的分片到达情况：

```python
# 在 main.py 或 .env 中
LOG_LEVEL=DEBUG
```

关键日志标签：
- `[UDP] device N frame M expired` → 有丢包，考虑增大 `CHUNK_DEADLINE_S`
- `[UDP] device N evict frame M` → pending 满，考虑增大 `MAX_PENDING`
- 无 `drain task started` → 设备 IP 未在数据库注册
